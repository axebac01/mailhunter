
The user wants to extend the existing mailhunter.ai app with a real Supabase backend, replacing the in-memory Zustand store with persisted tables, and add a more realistic import/export pipeline.

Current state: app is fully built with Zustand + mock data, all pages working. No Supabase yet.

Need to: enable Lovable Cloud, create 9 tables with proper schema/RLS, seed them with realistic data, refactor the store/pages to fetch from Supabase, and tighten the import + job simulation flows.

Key design decisions:
- No auth in MVP → tables need permissive RLS (this is an internal tool per the brief). I'll flag this clearly.
- Keep Zustand as a thin client cache layer; use TanStack Query for server state.
- Seed data via a SQL migration so the app feels populated on first load.
- Job simulation runs client-side on a timer (no edge function needed for MVP) — increments progress, appends logs, inserts contacts/people/source_pages.

Plan stays concise.

## Plan — Supabase-backed mailhunter.ai

### 1. Enable Lovable Cloud + create schema
Single migration creating all 9 tables with the exact fields from the brief, plus:
- Enums: `job_status`, `import_status`, `contact_type`, `crawl_log_level`, `source_type`, `export_type`, `file_format`, `page_type`
- Indexes on `companies.domain` (unique), `contacts(company_id, contact_type, value)` unique, `contact_people(company_id, full_name, role_title, department)` unique, plus FK indexes
- `updated_at` triggers
- RLS enabled on every table with permissive policies (`USING (true)`) since this is an internal no-auth MVP — I will surface this as an intentional choice and note it in Settings

### 2. Seed realistic data via migration
Insert ~12 jobs, ~40 companies, ~120 contacts (generic only), ~60 people, ~8 imports with rows, ~20 logs, ~5 exports. Filter out any person-tied email patterns at insert time.

### 3. Data layer refactor
- Add `src/integrations/supabase/` (auto-generated by Cloud)
- New `src/lib/api/` modules: `jobs.ts`, `companies.ts`, `contacts.ts`, `people.ts`, `imports.ts`, `logs.ts`, `exports.ts` — each exports typed query/mutation functions
- Wrap the app in `QueryClientProvider` (TanStack Query already a dep)
- Replace Zustand reads with `useQuery` hooks; mutations use `useMutation` + invalidation
- Keep a small Zustand slice only for transient UI state (filters, selection)

### 4. Import pipeline (real)
- Parse uploaded CSV/XLS/XLSX with `xlsx` (already installed)
- Show preview → column mapping UI → "Start import" 
- On start: insert `imports` row (status=processing) + `import_rows` rows (status=pending)
- Process rows client-side in batches: match by website→domain, dedupe, update row status (matched/partial_match/not_found/duplicate/failed), increment counters
- Surface live progress + final summary

### 5. Job simulation (real-feeling)
- "Start" on a job: set status=running, kick off a client-side ticker
- Every ~1.5s while running: increment progress, insert 1–2 `crawl_logs`, occasionally insert a `source_pages` row + a `contacts` row (generic email/phone/form) and sometimes a `contact_people` row, all respecting collection-scope toggles and dedupe constraints
- Personal-email regex filter applied before any contacts insert (`/^[a-z]+\.[a-z]+@/i` and similar) — discarded with a log entry
- Pause/Stop halts ticker; Resume restarts; Complete when progress hits maxCompanies or 100%

### 6. Export pipeline
- New `ExportButton` component with dropdown: All / Filtered / Selected × CSV / XLSX
- Generates file via existing `exporters.ts`, names per spec (`contacts_export_YYYY-MM-DD.csv` etc.)
- Inserts an `exports` row recording what was exported
- Strict allow-list of fields per the brief — no email field on people exports

### 7. Filter upgrades
Add the missing filters per the brief on Contacts (job, import, date range), People (company, role, department, job, import, date range), and Jobs (created date, last run date).

### 8. Component reusables
- `DataTableShell`, `FilterBar`, `ExportButton`, `KpiCard` (exists), `StatusBadge` (exists), `UploadZone`, `DetailPanel` — extract where pages currently inline these.

### 9. Settings additions
- "System status" panel reading live counts from Supabase
- "Reseed demo data" button → calls a SQL function (created in migration) that truncates + reseeds
- "Clear all data" with confirm

### Technical notes
- No auth means RLS is permissive; the security scanner will flag this. Acceptable for an internal MVP per the user's brief but I'll add a note in Settings + README.
- Personal-email filter is enforced in three places: import processing, job simulation, and a CHECK constraint on `contacts.value` (regex disallowing `firstname.lastname@` patterns) as defense in depth.
- All timestamps stored as `timestamptz`.
- Source pages and logs grow unbounded during simulation — cap logs per job at ~200 in the UI query.

### Out of scope
- Real crawling (simulated only, per brief)
- Auth (explicitly excluded by brief)
- Edge functions (not needed; all logic runs client-side against Supabase)
